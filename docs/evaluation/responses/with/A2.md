# A2 — How do I write tests for a component?

## Test types and locations

- **Unit tests**: `src/<component>/__tests__/` — run in JSDOM
- **Integration tests**: `src/<component>/__integ__/` — run in a real Chrome browser against dev pages
- **Motion tests**: `src/<component>/__motion__/` — like integration but with animations enabled

## Running tests

```sh
npm test                # all tests
npm run test:unit       # unit tests only
npm run test:integ      # integration tests (auto-starts dev server)
npm run test:a11y       # accessibility tests

# Target a specific file directly:
TZ=UTC npx jest -c jest.unit.config.js src/button/__tests__/button.test.tsx

# Integration (needs dev server running via `npm start`):
NODE_OPTIONS=--experimental-vm-modules npx jest -c jest.integ.config.js src/button/__integ__/

# Update snapshots:
TZ=UTC npx jest -u snapshot -c jest.unit.config.js src/
```

## Unit test patterns

Use `render` from `@testing-library/react` + `createWrapper` from test-utils for querying:

```tsx
import { render } from '@testing-library/react';
import Button from '../../../lib/components/button';
import createWrapper from '../../../lib/components/test-utils/dom';

function renderButton(props = {}) {
  const { container } = render(<Button {...props} />);
  return createWrapper(container).findButton()!;
}

test('renders disabled state', () => {
  const wrapper = renderButton({ disabled: true });
  expect(wrapper.isDisabled()).toBe(true);
  expect(wrapper.getElement()).toHaveClass(styles.disabled);
});
```

Key points:
- Prefer test-utils wrappers (`createWrapper(...).findButton()`, `.findInput()`, etc.) over raw DOM queries — they provide typed, component-aware methods like `.isDisabled()`, `.getElement()`, `.findAll()`.
- Import wrappers from `test-utils/dom` for unit tests.
- Use `jest.fn()` for spies, `fireEvent` / `act` from testing-library for interactions.

## Integration test patterns

Use `useBrowser` + page objects + `createWrapper` from `test-utils/selectors` (not dom):

```ts
import useBrowser from '@cloudscape-design/browser-test-tools/use-browser';
import { BasePageObject } from '@cloudscape-design/browser-test-tools/page-objects';
import createWrapper from '../../../lib/components/test-utils/selectors';

const wrapper = createWrapper();

test('tab skips disabled buttons', useBrowser(async browser => {
  await browser.url('#/light/button/tab-navigation');
  const page = new BasePageObject(browser);
  await page.click(wrapper.findButton('#focusButton').toSelector());
  await page.keys('Tab');
  await expect(page.getFocusedElementText()).resolves.toBe('Active with href');
}));
```

Key difference: `test-utils/selectors` generates CSS selector strings for browser automation, while `test-utils/dom` operates directly on DOM nodes for JSDOM unit tests.

Integration tests run against dev pages located in `pages/<component>/`, so you need a corresponding page for your test scenario.

## Follow-up: What about visual regression tests?

### How they run

Visual regression tests run automatically when you open a PR in GitHub. There's no local command — they're CI-only. The GitHub Action takes screenshots of permutation pages and compares them against baselines.

To check results:
1. Look at the "Visual Regression Tests" action in your PR
2. The "Test for regressions" step logs which pages failed
3. Download the `visual-regression-results` artifact from the action summary and open `html_report/index.html` for a full visual diff

If changes are expected, call it out in your PR comments.

### Writing permutation pages

Permutation pages live at `pages/<component>/permutations.page.tsx`. They render every meaningful combination of props so the screenshot captures all visual states. The key utilities:

- `createPermutations<Props>([...])` — generates the cartesian product of prop values you provide
- `PermutationsView` — renders each permutation
- `ScreenshotArea` — wraps content to define the screenshot boundary (use `disableAnimations={true}`)

Example for a component:

```tsx
import MyComponent, { MyComponentProps } from '~components/my-component';
import createPermutations from '../utils/permutations';
import PermutationsView from '../utils/permutations-view';
import ScreenshotArea from '../utils/screenshot-area';

const permutations = createPermutations<MyComponentProps>([
  {
    variant: ['default', 'primary'],
    disabled: [false, true],
    children: ['Label'],
  },
]);

export default function MyComponentPermutations() {
  return (
    <>
      <h1>MyComponent permutations</h1>
      <ScreenshotArea disableAnimations={true}>
        <PermutationsView permutations={permutations} render={permutation => <MyComponent {...permutation} />} />
      </ScreenshotArea>
    </>
  );
}
```

Each object in the array passed to `createPermutations` produces a separate cartesian product. Use multiple objects when certain prop combinations only make sense together (e.g., `variant: ['icon']` only with `ariaLabel`, not `children`).
