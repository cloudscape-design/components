# A4 — How is a component organized in this repo?

## Prompt

> How is a component organized in this repo? What should I consider when implementing a new component?

## Response

### Component Organization

Each component in this repo follows a consistent structure under `src/<component-name>/`:

```
src/badge/                          # Simple component example
├── index.tsx                       # Public API entry point
├── interfaces.ts                   # TypeScript props interface (extends BaseComponentProps)
├── styles.scss                     # Component styles (uses design tokens)
├── style.tsx                       # Style API logic (runtime style customization)
├── __tests__/
│   └── badge.test.tsx              # Unit tests
```

Larger components add more files:

```
src/button/                         # Complex component example
├── index.tsx                       # Public wrapper (delegates to internal.tsx)
├── internal.tsx                    # Actual implementation (used by other internal components)
├── interfaces.ts                   # Public + internal prop types
├── styles.scss                     # Main styles
├── constants.scss                  # SCSS constants/variables
├── style.tsx                       # Style API logic
├── icon-helper.tsx                 # Sub-components / helpers
├── test-classes/styles.scss        # CSS selectors exposed for test-utils
├── analytics-metadata/             # Analytics integration
├── __tests__/                      # Unit tests
└── __integ__/                      # Integration tests
```

Beyond `src/`, each component also has:

- `pages/<component>/` — Demo/permutation pages for visual testing (e.g., `permutations.page.tsx`)
- `src/test-utils/dom/<component>/index.ts` — Test wrapper class (extends `ComponentWrapper`)

### Key Patterns to Follow When Implementing a New Component

1. **index.tsx is the public entry point** — it calls `useBaseComponent('ComponentName')` for telemetry, extracts base props via `getBaseProps()`, and calls `applyDisplayName()`. Simple components render directly here; complex ones delegate to `internal.tsx`.

2. **internal.tsx (optional)** — used when other internal components need to consume your component with extended props (e.g., `InternalButton` has extra variants like `'flashbar-icon'` not exposed publicly).

3. **interfaces.ts** — define your props interface extending `BaseComponentProps`. Use JSDoc comments (they feed the documentation site). The `@awsuiSystem core` annotation marks features gated to the core system.

4. **styles.scss** — use `@use '../internal/styles'` for resets and mixins, and `@use '../internal/styles/tokens' as awsui` for design tokens. Never use hardcoded colors/spacing.

5. **style.tsx** — if your component supports the Style API, implement a `get<Component>Styles()` function that maps style prop values to CSS properties, gated behind `SYSTEM === 'core'`.

6. **test-classes/styles.scss** — expose CSS class selectors that test-utils can use to find elements (separate from visual styles).

7. **src/test-utils/dom/<component>/index.ts** — create a wrapper class extending `ComponentWrapper` with a `rootSelector` pointing to your root CSS class. Add finder methods for sub-elements.

8. **pages/<component>/permutations.page.tsx** — create visual permutation pages that render all prop combinations for screenshot testing.

9. **__tests__/** — unit tests using `@testing-library/react` + the test-utils wrapper (`createWrapper`). Import from `../../../lib/components/...` (the build output path).

### Other Considerations

- Use `'use client'` directive at the top of `index.tsx` for Next.js compatibility
- Leverage shared internal infrastructure: hooks (`src/internal/hooks/`), contexts (`src/internal/context/`), shared components (`src/internal/components/`), and utilities (`src/internal/utils/`)
- Use `WithNativeAttributes` for the root element to support the `nativeAttributes` prop pattern
- Use `warnOnce` from `@cloudscape-design/component-toolkit/internal` for dev-mode warnings about invalid prop combinations
- Consider analytics integration (useFunnel, performance marks) if the component is interactive
