# A2 — How do I write tests for a component?

## Prompt

> How do I write tests for a component? What patterns and utilities should I use? How do I run tests?

## Response

Here's a comprehensive overview of how testing works in this Cloudscape components repo:

### Test Types

There are 4 test suites, each with its own Jest config:

| Suite | Directory | Config | Environment | Purpose |
|-------|-----------|--------|-------------|---------|
| Unit | `src/<component>/__tests__/` | `jest.unit.config.js` | jsdom | Component rendering, props, events, accessibility |
| Integration | `src/<component>/__integ__/` | `jest.integ.config.js` | node | Browser-based tests via `@cloudscape-design/browser-test-tools` |
| A11y | `src/__a11y__/` | `jest.integ.config.js` | node | Accessibility audits |
| Motion | `src/<component>/__motion__/` | `jest.motion.config.js` | node | Animation/transition tests |

### Running Tests

```sh
# All tests
npm test

# Just unit tests
npm run test:unit

# Run a single component's unit tests
npx jest --config jest.unit.config.js src/button/__tests__/button.test.tsx

# Integration tests (requires a build first)
npm run test:integ

# A11y tests
npm run test:a11y

# Motion tests
npm run test:motion
```

### Unit Test Patterns

Unit tests use `@testing-library/react` + `@testing-library/jest-dom` with Cloudscape's own test-utils wrappers.

The core pattern:

```tsx
import React from 'react';
import { render } from '@testing-library/react';
import createWrapper from '../../../lib/components/test-utils/dom';
import Button, { ButtonProps } from '../../../lib/components/button';

function renderButton(props: ButtonProps = {}) {
  const renderResult = render(<Button {...props} />);
  return createWrapper(renderResult.container).findButton()!;
}

test('renders disabled state', () => {
  const wrapper = renderButton({ disabled: true });
  expect(wrapper.isDisabled()).toBe(true);
  expect(wrapper.getElement()).toHaveClass(styles.disabled);
});
```

Key utilities:

- `createWrapper(container)` — from `lib/components/test-utils/dom` — returns a wrapper with typed `find*` methods for each component (e.g., `findButton()`, `findToken()`, `findDismiss()`)
- Each component has a test-utils wrapper class in `src/test-utils/dom/<component>/` that exposes semantic finders like `findLabel()`, `findDismiss()`, `findLoadingIndicator()`, etc.
- CSS class assertions use imported styles objects: `import styles from '../../../lib/components/button/styles.css.js'`
- `jest.fn()` for spy/mock callbacks
- `fireEvent` from `@testing-library/react` for mouse/keyboard events
- `act()` for state updates
- `jest.spyOn(console, 'warn')` to assert dev warnings
- `@testing-library/jest-dom` matchers like `toHaveClass`, `toHaveAttribute`, `toHaveTextContent`, `toHaveAccessibleName`, `toBeInTheDocument`

### Integration Test Patterns

Integration tests use `@cloudscape-design/browser-test-tools` with real browser automation:

```ts
import { BasePageObject } from '@cloudscape-design/browser-test-tools/page-objects';
import useBrowser from '@cloudscape-design/browser-test-tools/use-browser';
import createWrapper from '../../../lib/components/test-utils/selectors'; // note: selectors, not dom

class MyPage extends BasePageObject {
  getTokenWrapper(selector: string) {
    return createWrapper().findToken(selector);
  }
}

function setupTest(testFn: (page: MyPage) => Promise<void>) {
  return useBrowser(async browser => {
    await browser.url('#/light/token/simple');
    const page = new MyPage(browser);
    await page.waitForVisible('h1');
    await testFn(page);
  });
}

test('displays label', setupTest(async page => {
  const label = page.getTokenWrapper('[data-testid="my-token"]').findLabel();
  expect(await page.getText(label.toSelector())).toBe('Expected text');
}));
```

Key difference: integration tests import from `test-utils/selectors` (CSS selector strings for browser queries) instead of `test-utils/dom` (DOM wrappers for jsdom).

### File Structure for a Component's Tests

```
src/my-component/
├── __tests__/
│   ├── my-component.test.tsx    # unit tests
│   ├── styles.test.tsx          # CSS/style tests
│   └── analytics-metadata.test.tsx
├── __integ__/
│   └── my-component.test.ts    # browser integration tests
└── __motion__/
    └── my-component.test.ts    # animation tests
```

## Follow-up Response (visual regression / browser testing)

This codebase doesn't use traditional pixel-diff visual regression testing (like Percy or Chromatic) directly in the repo. Instead, it uses a permutations-based screenshot page pattern. Here's how it works:

### Visual Regression Testing Pattern

Visual regression is done through permutation pages — dedicated pages that render a component in every meaningful combination of props, wrapped in a `ScreenshotArea`. An external screenshot tool captures these pages and diffs them against baselines.

#### The 3 building blocks:

1. `createPermutations<Props>(...)` — generates the cartesian product of all prop combinations
2. `PermutationsView` — renders each permutation with its props serialized as a `data-permutation` attribute
3. `ScreenshotArea` — marks the region to be captured, optionally disabling animations

#### How to write one

Create a file at `pages/<component>/permutations.page.tsx`:

```tsx
import React from 'react';
import Button, { ButtonProps } from '~components/button';
import createPermutations from '../utils/permutations';
import PermutationsView from '../utils/permutations-view';
import ScreenshotArea from '../utils/screenshot-area';

const permutations = createPermutations<ButtonProps>([
  {
    disabled: [false, true],
    variant: ['normal', 'primary'],
    children: ['Click me'],
    iconName: [undefined, 'settings'],
  },
]);

export default function ButtonPermutations() {
  return (
    <>
      <h1>Button permutations</h1>
      <ScreenshotArea disableAnimations={true}>
        <PermutationsView
          permutations={permutations}
          render={permutation => <Button {...permutation} />}
        />
      </ScreenshotArea>
    </>
  );
}
```

#### Key details:

- `createPermutations` takes an array of objects where each key maps to an array of possible values. It generates every combination (cartesian product). You can pass multiple objects to create separate groups of permutations.
- There's a hard cap of 276 permutations per page. If you exceed it, split into multiple pages.
- `ScreenshotArea` has a `disableAnimations` prop — use it to avoid flaky diffs from CSS transitions.
- The page height limit is 16,000px. If exceeded, a warning is rendered and the test will fail.
- Pages are served at routes like `#/light/button/permutations` (derived from the file path under `pages/`).

#### Viewing pages locally:

```sh
npm run start        # serves pages at localhost
npm run start:integ  # serves integ-specific pages
```

Then navigate to `#/light/<component>/permutations` in the browser.

The actual screenshot capture and diffing happens in the CI pipeline via `@cloudscape-design/browser-test-tools`, not as a test you run locally with a single command.
